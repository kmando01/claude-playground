# Skill Feedback Loop

스킬이 만든 산출물(ADR, 코드, 문서 등)에 대한 피드백을 받았을 때, 일반화 가능한 부분을 원본 스킬에 반영하는 워크플로우.

---

## 자동 트리거 — 사용자가 묻지 않아도 능동적으로 발동

다음 중 하나가 발생하면, 사용자가 "스킬에 반영해줘"라고 말하지 않아도 스킬을 업데이트한다.

### 1. 버그 발견 + 수정
세션 중 스킬에 문서화된 패턴/스크립트/명령에서 버그가 발견되고 수정됐다면:
- 수정된 패턴을 스킬에 즉시 반영
- 스킬에 `## ⚠️ 알려진 버그 & 필수 패치` 섹션이 없으면 추가

예시:
- k6 `res.status===200` 발급 집계 버그 발견 → `perf-bench`, `perf-test-reference` 업데이트
- mysql snapshot pipe+heredoc 충돌 발견 → `perf-bench` 업데이트
- MongoDB 어댑터 공정성 문제 발견 → `perf-bench` 업데이트

### 2. 스킬 명세와 현실 불일치
스킬에 명시된 경로/프로파일/명령이 코드베이스 실제와 다를 때:
- 스킬 명세를 현실에 맞게 수정

예시: 멀티모듈 전환으로 JAR 경로 변경 → `perf-bench` Step 2 경로 수정

### 3. 중요 학습 패턴 — 다음 세션에서 반드시 알아야 할 것
세션 전반에 걸쳐 반복적으로 중요했던 패턴이 스킬에 없다면 추가:
- "sold-out flag 유무가 모든 어댑터의 TPS 천장을 결정한다"
- "bench-mongo-full vs bench-mongo 공정성 구분"

**자동 트리거 판단 기준**: "다음 번에 같은 스킬을 쓰면 이 문제가 또 발생하는가?" → YES → 지금 반영

## 명시적 트리거 조건

- 사용자가 명시적으로 "스킬에 반영", "/skill-feedback" 등 요청
- 스킬 산출물에 피드백을 적용한 뒤, 사용자가 "이거 스킬에도 적용해줘" 요청

## 실행 절차

### Step 1: 피드백 분석

대화에서 다음을 추출한다:

| 항목 | 설명 | 필수 |
|------|------|------|
| **원본 스킬** | 산출물을 만든 스킬 (추론 또는 사용자 확인) | ✅ |
| **피드백 내용** | 사용자가 지적한 문제 또는 개선점 | ✅ |
| **산출물 변경** | 피드백을 반영해서 산출물에 실제 적용한 수정 | ✅ |
| **일반화 가능성** | 이 피드백이 이 산출물만의 문제인지, 모든 산출물에 적용할 원칙인지 | ✅ |

### Step 2: 일반화 판단

**스킬에 반영해야 하는 피드백** (일반화 가능):
- 구조/포맷에 대한 피드백 (예: "Risks를 성격별로 나눠라")
- 누락된 관점 (예: "레이스 컨디션 시나리오를 항상 적어라")
- 리뷰어 관점의 가독성 개선 (예: "비교표에서 선택안 약점도 동등하게")

**스킬에 반영하지 않는 피드백** (일회성):
- 이 산출물의 특정 내용에 대한 수정 (예: "TTL을 1분이 아니라 5분으로")
- 도메인 특수 사항 (예: "알림 시스템에서는 상태 지연이 OK")
- 오타 수정

판단이 애매하면 사용자에게 확인:
> "이 피드백을 앞으로 모든 {산출물 유형}에 적용할 일반 원칙으로 스킬에 반영할까요, 아니면 이번 산출물만의 수정인가요?"

### Step 3: 원본 스킬 읽기

```bash
# 스킬 파일 찾기 (파일 또는 디렉토리)
ls -la .claude/skills/{skill-name} 2>/dev/null || ls -la .claude/skills/{skill-name}/SKILL.md 2>/dev/null
```

스킬을 읽고 다음을 파악한다:
- 기존 원칙/체크리스트 구조
- 피드백이 들어갈 위치 (원칙 섹션? 템플릿? 체크리스트?)
- 기존 원칙과 충돌하는 부분이 있는지

### Step 4: 스킬 수정

피드백을 스킬에 반영할 때 3곳을 점검한다:

1. **원칙 섹션**: 새로운 MUST/SHOULD 원칙 추가 또는 기존 원칙 보강
2. **템플릿**: 템플릿 구조가 바뀌어야 하면 업데이트
3. **체크리스트**: 검증 항목 추가

수정 시 주의사항:
- 피드백의 **"왜"**를 함께 기록 (예: "리뷰어가 어디부터 봐야 할지 모름")
- 기존 원칙 번호를 유지 (새 원칙은 뒤에 추가)
- 원칙 헤더에 출처 표기 제거 — 원칙은 출처와 무관하게 독립적이어야 함
  - BAD: `### 10. Risks 구조화 (ADR-004 피드백에서 도출)`
  - GOOD: `### 10. Risks는 성격별로 나누고, Implementation Plan과 연결`

### Step 5: 검증

```
□ 스킬 원칙 섹션에 피드백이 일반 원칙으로 반영됐는가?
□ 템플릿이 원칙과 일관성 있게 업데이트됐는가?
□ 체크리스트에 검증 항목이 추가됐는가?
□ 기존 원칙과 충돌이 없는가?
□ 피드백의 "왜"가 함께 기록됐는가?
□ 출처 특정 문구(예: "ADR-004에서 도출")가 제거됐는가?
```

### Step 6: CE 헬스체크 (스킬이 rule에도 영향을 주면)

스킬 수정이 대응하는 rule에도 영향을 주는 경우:
- rule도 함께 업데이트 (30줄 제한 유지)
- `wc -l .claude/rules/*.md | tail -1`로 총량 확인

## 예시 흐름

```
사용자: [ADR-004에 피드백] "Risks가 8개로 늘어나면서 리뷰어가 어디부터 봐야 할지 모름"
→ 산출물 수정: ADR-004 Risks를 "구현 시 검증 필수" / "인지 사항"으로 분리
→ 사용자: "이것도 스킬에 반영"

Step 1: 원본 스킬 = adr-template, 피드백 = Risks 가독성
Step 2: 일반화 가능 ✅ (모든 ADR에 적용할 구조 원칙)
Step 3: adr-template 읽기 → 원칙 10개 + 템플릿 + 체크리스트
Step 4: 원칙 10번 보강 + 템플릿 Risks 구조 변경 + 체크리스트 항목 추가
Step 5: 검증 통과
```
