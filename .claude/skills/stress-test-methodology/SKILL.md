# 성능/스트레스 테스트 방법론

> **핵심 메시지**: 스트레스 테스트는 "부하를 거는 일"이 아니라 **"의사결정의 근거를 만드는 일"**이다.
> 결과물이 "문제없습니다" 한 줄이라면 그건 테스트가 아니라 단순 부하 확인이다.

---

## 0. 테스트 목적 두 가지

1. **변경 전후를 수치로 비교** — "이 변경을 적용해도 괜찮은가?"에 답하기 위해
2. **시스템이 죽기 직전 양상 파악** — 병목·임계치를 미리 알아 운영 시 빠른 대응을 위해

## 1. 환경 구성 철칙

### 1.1 전제: 테스트 환경 ≠ 운영 환경

완벽히 동일한 환경은 없다. 이를 인정하지 않으면 잘못된 결론에 도달한다.

| 반드시 일치시킬 것 | 일치시킬 수 없다고 인정할 것 |
|---|---|
| JVM heap, CPU/메모리 limit, GC 설정 | 서버 대수, 외부 의존성 실부하 |
| 인스턴스 단위 스펙 | 네트워크 토폴로지, 클라이언트 분포 |
| — | 운영 데이터 규모 및 분포 |

**인스턴스 스펙마저 다르면 측정값은 의미를 잃는다.**

### 1.2 측정값 해석: 대당(per-instance) 수치 기준

```
대당 RPS × 운영 서버 대수 = 예상 총 RPS
대당 CPU × 운영 대수 / 새 인프라 코어 수 = 예상 인프라 증감
```

**주의**: 이 환산은 선형성을 가정한 추정이다. 서버 대수 증가로 DB 커넥션 경합, LB 오버헤드 등 비선형 요인이 생긴다.
리포트에 환산 수치를 적을 때는 **가정(assumption)도 반드시 명시**한다.

### 1.3 워밍업 원칙 — 테스트할 경로를 직접 소화해야 한다 ⚠️

**핵심**: 워밍업은 "부하를 좀 주는 것"이 아니라 **"측정할 코드 경로를 JIT threshold 이상 실행시키는 것"**이다.

#### 흔한 함정 — sold-out만 타는 워밍업

limit=1,000인 선착순 쿠폰 테스트 예시:
- 워밍업 1,000 RPS 30s → 발급 1,000건 완료 → 이후 ~24,000건 전부 sold-out(거절)
- **발급 성공 경로(INSERT × 2)는 겨우 1,000번만 실행**됨
- JVM JIT threshold(메서드당 약 10,000~15,000회) 미달 → INSERT 경로 미컴파일 상태
- 실제 측정 첫 라운드에서 그 경로가 처음으로 제대로 실행 → **Cold start 노이즈**

**실제 사례 (2026-06-15 kakaopay-coupon 어댑터 비교)**:
- 기존 워밍업: MongoDB 3K P95(발급) = **198ms**
- 발급 경로 3사이클 풀소화 후: **4~8ms** (25배 차이)

#### 올바른 워밍업 패턴

```
사이클 1~3 반복:
  reset → 부하 → limit 소진(INSERT 완료) 확인 → 다음 사이클
```

기준: **측정할 저장소 연산(INSERT / findAndModify / INCR)이 사이클당 limit 만큼 실행** → 3사이클이면 JIT 충분.

#### 경로별 체크

| 어댑터 유형 | 워밍업 시 반드시 태워야 할 경로 |
|---|---|
| MySQL INSERT 기반 | coupon_issues INSERT + event_participations INSERT |
| MongoDB 풀패스 | findAndModify + coupon_issues INSERT + event_participations INSERT |
| Redis 이중 카운터 | INCR × 2 + MySQL INSERT (sold-out 경로는 INCR만 — 별도 소화 불필요) |
| SKIP LOCKED | coupon_pool UPDATE + coupon_issues INSERT |

**판정**: 사이클 1 avg latency >> 사이클 2 avg latency 이면 워밍업 미완. 사이클 간 수렴할 때까지 반복.

### 1.4 시나리오는 파라미터 조합으로 관리

새 시나리오마다 코드를 새로 짜지 말고:
- **Task**: 어떤 API가 어떤 비율로 호출되는지
- **TestProperties**: 데이터 분포 (어댑터, RPS 등)
- **target**: 어떤 엔드포인트에 부하를 줄지

→ "평시 / 피크 / 신규 기능 / 최악 케이스"를 빠르게 조합 가능.

## 2. 5가지 핵심 원칙

### 원칙 1. 측정 목적을 한 문장으로

```
"이번 테스트의 목적은 _______ 이고, 성공 기준은 _______ 이다."
```

이 문장이 안 쓰이면 테스트 시작 금지. 목적 카테고리:
- 성능 부족 해결 / 유지보수·확장성 개선 / 신기술 도입 검증 / 보안·규정 대응

### 원칙 2. 변경 전 수치 먼저

**개편 후가 좋다는 것을 보이려면 개편 전 수치가 먼저 있어야 한다.**

- [ ] 대당 / 총 RPS · CPU · Memory (평시 / 피크)
- [ ] 비즈니스 로직 부하 vs 프로토콜 부하 비중
- [ ] 외부 의존성 응답 시간 분포

### 원칙 3. 후보군을 합리적 기준으로 좁혀라

여러 후보를 다 돌려보는 건 함정. 변수가 많아져 해석이 무의미해진다.
판단 기준: 커넥션 특성 부합 여부 / 팀 운영 친숙도 / EOL·라이선스

### 원칙 4. 한 라운드 = 한 변수

통일해야 하는 것: TPS · 부하 패턴 · 지속 시간 · 측정 시각 · JVM 설정 · 서버 커밋
**그래프에 단위/시간/환경 정보 없으면 며칠 후 해석 불가.**

### 원칙 5. 숫자로 정제하라

```
| 항목 | 변경 전 | 변경 후 | 의사결정 |
|---|---|---|---|
| 대당 평균 CPU | 60% | 65% (+5%p) | 인프라 약 10대 증가 필요 |
```

한 대의 +5%p가 운영 120대 기준으로는 인프라 +10대 — 이 환산이 정제의 핵심.

## 3. 레이어 분석 — 어디까지 내려갈 것인가

바깥 → 안쪽 순서로 내려간다. 이상 없으면 다음 레이어 내려갈 필요 없다.

### Layer 1. Endpoint 지표 (항상 먼저 본다)

| 지표 | 비정상 신호 |
|---|---|
| RPS | 예상보다 낮은 TPS에서 포화, 비선형 튐 |
| Latency (P50/P95/P99) | P95·P99 급등 → 처리량 한계 / P50도 길어지면 전반적 문제 |
| Error rate | 5xx → 서버 한계 / timeout → 클라이언트 자원 부족 / 4xx → 시나리오 데이터 이상 |

### Layer 2. 시스템 자원

- CPU: process load / system load → 스레드·CPU 프로파일 연결
- Memory: RSS / 페이지 캐시 → JVM heap 연결
- I/O: 네트워크 I/O, 파일 디스크립터
- 네트워크 대역폭 한계 도달 여부

### Layer 3. JVM/런타임 상태

| 지표 | 무엇을 알 수 있나 |
|---|---|
| Thread state (blocked/waiting) | blocked 증가 → 락 경합 / waiting 증가 → 외부 호출 대기 |
| Executor queue | queue 차오름 → 처리 속도 < 입력 속도 |
| GC pause (count/sum/max) | allocation rate 폭증 → 일시 객체 문제 |
| Heap 사용량 추세 | Old Gen 우상향 + GC 후 안 내려옴 → 회수 불가 객체 |

**지표는 단독이 아닌 연쇄 인과 관계로 본다:**
```
heap 점유 → ZGC pause 못 따라감 → CPU load 상승 → executor queue 적체 → readTimeout
```

### Layer 4. 객체 레벨 (디버깅 단계)

L3까지 봤는데 답이 없거나 heap이 GC 후에도 안 내려올 때 진입.

| 도구 | 용도 |
|---|---|
| Heap dump / `jmap -histo` | 어떤 객체가 어느 GC root에서 살아남는지 |
| CPU profiling (JFR, async-profiler) | flame graph로 메서드 CPU 점유 |
| Thread dump (`jstack`) | blocked/waiting 스레드 스택 → 락 경합 지점 |

> L4에 도달하면 더 이상 측정이 아니라 **디버깅**이다. PR로 이어지는 단계.

### 레이어 결정 테이블

| 측정 결과 | 다음 액션 |
|---|---|
| L1 Endpoint 정상 | 종료 |
| L2 자원 한계 | 인프라 스케일 업/아웃 |
| L3 JVM 한계 | JVM/런타임 튜닝 |
| L4 객체 문제 | 코드 픽스 PR |
| 어디에도 안 잡힘 | 추가 지표 필요 여부 재검토 |

## 4. 자주 발생하는 테스트 케이스

### Case A. 관측성(Observability) 인프라 추가
- [ ] 애플리케이션 처리량/응답 시간 손실 없는가
- [ ] 호스트 CPU/메모리/네트워크 추가 부하 측정
- [ ] 고부하 시 내부 지표 지연/알림 지연 발생하는가

### Case B. 프레임워크/프로토콜 벤치마크
- [ ] IO sleep만 가정한 상태에서 부하 비교 (프레임워크 자체 특성)
- [ ] 워커 수를 늘리며 RPS당 latency 곡선 비교

### Case C. 인프라 OS 변경 / 보안 시스템 추가
- [ ] 평시뿐 아니라 피크 트래픽에서도 영향 측정
- [ ] slab 메모리 누수, 백신 동작 시 리소스 스파이크

### Case D. 도메인 특화 최악 케이스
- 선착순 쿠폰: 동시 N만 명이 동일 시점에 POST
- 채팅: 한 채팅방 수백 명 동시 write / 자정 burst
→ **신규 기능 런칭 전에 반드시 모의 테스트**

## 5. AI 자동화 패턴

> **판단은 사람이, 반복은 LLM이.** 자동화의 목표는 분석에 더 많은 시간을 쓰는 것.

### 5.1 스킬 단위 분리 (Claude Code 기준)

| 스킬 | 역할 |
|---|---|
| `/deploy-target` | 빌드 → 배포 → 헬스체크 |
| `/run-stress` | k6 실행 (ADAPTER, TARGET_RPS 파라미터화) |
| `/capture-metrics` | Prometheus 쿼리 + Grafana 패널 캡처 (subagent 병렬) |
| `/make-report` | 캡처 결과 → REPORT.md 생성 |

### 5.2 Subagent 병렬 캡처 패턴

지표 수집이 무거울 때 Layer별로 subagent 분리:
- Agent 1: L1 Endpoint 지표 + Grafana TPS/Latency/에러 패널
- Agent 2: L2 자원 지표 + Grafana HikariCP/Tomcat 패널
- Agent 3: L3 JVM 지표 + Grafana Heap/GC + 어댑터별 패널

각 agent가 Prometheus 쿼리 → Grafana 캡처 → 요약까지 병렬 수행.
메인 컨텍스트는 결론만 받아 REPORT.md 작성.

### 5.3 Background Bash + Monitor 패턴

긴 테스트(30분+)는 background bash로 띄우고 Monitor로 진행 추적:
```bash
# background로 k6 실행, 로그 파일에 이벤트 발행
bash perf-test/run-bench.sh > /tmp/bench.log 2>&1 &
# Monitor로 로그 추적 → 완료 이벤트 탐지 → 메인에 알림
```

### 5.4 AI에게 맡길 수 없는 것

**자동화된 리포트는 그럴듯하게 거짓말을 한다.** 실제 사례:
- 리포트 실패율 0%인데 서버 에러 카운트 증가 → 클라이언트 코드 검증 누락
- 비교군/대조군 결과가 같음 → 이미지 캐시로 같은 버전 배포됨. AI는 "성능 매우 좋음!" 기재

**대응 원칙:**
- 성공/실패/검증 기준을 LLM에 명시적으로 알린다
- 배포 이미지 다이제스트 확인 등 환경 검증을 스킬에 포함
- 명확한 목표치가 있으면 AI 결과를 한 번 의심하고 교차 확인
- 방법론적 통찰("두 서버를 같은 user 수로 비교하지 말고 곡선 자체를 뽑아라")은 AI가 주지 않는다

## 6. 테스트 전후 체크리스트

### 시작 전
- [ ] 목적이 한 문장으로 정리됨
- [ ] 테스트 환경과 운영의 차이점 명시됨 (서버 대수, 외부 의존성 등)
- [ ] 인스턴스 단위 스펙이 운영과 일치함
- [ ] 변경 전 지표가 평시/피크 모두 정리됨
- [ ] 통제할 변수 목록이 명확함

### 진행 중
- [ ] 한 라운드에 한 변수만 변경
- [ ] 워커 수를 점진적으로 증가시키며 곡선을 뽑음
- [ ] 클라이언트가 병목이 아닌지 확인됨
- [ ] **워밍업이 측정할 코드 경로를 직접 소화했는가** (→ 1.3 참조)
- [ ] 사이클 간 avg latency가 수렴했는가 (수렴 전 측정 = Cold start 노이즈)

### 결과 분석
- [ ] L1 지표가 정상인가 → 이상 시 L2→L3→L4 순서로 내려감
- [ ] 단독 지표가 아닌 연쇄 인과 관계로 해석했는가

### 리포트 정제
- [ ] 그래프에 단위/시간/환경 정보 명시됨
- [ ] 대당 수치를 운영 규모로 환산한 결과 포함됨
- [ ] 환산 시 사용한 가정(선형성 등)과 한계가 명시됨
- [ ] "문제없음"이 아닌 숫자로 결론이 뒷받침됨
