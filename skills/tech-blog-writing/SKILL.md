# 기술 블로그 작성 스킬

> **수치로 말하고, 실패를 숨기지 않으며, 버린 선택지를 적는다. 그 모든 판단이 왜 사용자에게 닿는지까지 쓴다.**
>
> 이 스킬은 [toss/technical-writing 가이드](https://github.com/toss/technical-writing)를 기반으로 구성됩니다.

---

## Step 0. 문서 유형 먼저 정하기

글을 쓰기 전에 어떤 목적의 글인지 먼저 정한다. 유형마다 구조와 강조점이 달라진다.

| 유형 | 목적 | 기술 블로그에서의 예 |
|------|------|-------------------|
| **깊은 이해** | 개념·원리·철학을 납득시킴 | 새 아키텍처 결정 이유, 기술 선택 배경 |
| **문제 해결** | 독자가 비슷한 문제를 풀 수 있게 도움 | 트러블슈팅, How-to 가이드, 장애 회고 |
| **학습** | 단계별로 따라하며 새 기술·도구를 익힘 | 내부 도구 온보딩, 실습 가이드 |
| **참조** | 정확하고 완전한 정보를 빠르게 찾음 | API 변경 공지, 설정 레퍼런스 |

기술 블로그 대부분은 **"깊은 이해"** + **"문제 해결"** 복합형이다.

## Step 1. 정보 구조 설계 (Architecture)

### 가치를 먼저 제공하기

독자는 "이 글을 읽으면 무엇을 얻는가?"를 먼저 알고 싶어 한다.

- 배경 설명보다 독자가 얻을 가치를 **첫 문단**에 명시
- "이 API는 여러 설정 옵션을 제공합니다" X → "이 API를 쓰면 로그를 실시간으로 처리할 수 있습니다" O
- 기능 나열이 아닌 **해결하는 문제와 변화**를 먼저

### 개요 빠트리지 않기

- 제목 바로 아래, 본문 전에 **핵심 내용 2~3문장** 요약
- "이 글을 읽으면 무엇을 할 수 있는가"에 대한 답을 개요에서 제공
- 기술적 배경 설명보다 핵심 정보를 먼저

### 효과적인 제목 쓰기

- 핵심 키워드를 제목에 포함 (독자가 검색할 단어)
- 30자 이내, 평서문으로 작성 (느낌표·물음표 금지)
- 소제목은 동일한 패턴으로 일관성 유지 (~하기 / ~하는 방법 등)
- 단계가 있으면 번호를 포함

### 한 페이지에서는 하나만 다루기

- H4(`####`) 이상으로 깊어지면 문서를 나눠야 한다는 신호
- 두 개 이상의 핵심 주제가 한 글에 있으면 분리 검토
- 넓은 주제를 다뤄야 한다면 개요 페이지 → 각 개념별 페이지로 연결

### 예측 가능하게 하기

- 일관된 섹션 이름과 순서 유지 (한 글에서 "설치"와 "환경 설정"을 혼용하지 않음)
- 코드 블록 앞에 항상 설명 먼저, 코드는 그 다음
- 기본 개념 → 구체적 사용법 → 예제 → 심화 순서로 배치
- 용어는 일관되게 — 한 글 안에서 "상태", "데이터", "값"을 같은 개념에 혼용하지 않음

### 자세히 설명하기

- 새 개념이 등장하면 한두 문장으로 즉시 정의
- 기능의 동작 조건, 입력/출력, 단위까지 명시
- 독자가 "배경 없이 추론해야 하는 부분"을 없앨 것

## Step 2. 글의 5단 구조 (기술 블로그 특화)

### ① 훅(Hook)

첫 1~3문장. 코드·아키텍처 용어 금지. 두 패턴 중 하나:
- **일상 경험**: "사용자가 '결제하기'를 누르는 순간, 보이지 않는 곳에서..."
- **규모로 임팩트**: "피크 시간대 동시 접속자 수는 분당 수십만 명..."

질문형으로 독자를 끌어들인다: "~해보셨을 겁니다", "~일까요?"

### ② 배경(Context)

팀·프로젝트 맥락. **구체적 숫자 1개 이상** 필수 (연도, 규모, 팀 사이즈).  
도메인 용어는 처음 등장 시 한 줄로 풀어준다. 교과서식 설명이 아니라 이야기 흐름에 녹인다.

### ③ 문제 정의(Problem)

추상적 묘사 금지. 구체적 고통을 보여준다:
- 실제 코드·구조·데이터로 보여줌
- 결과적 피해 명시 ("신규 입사자 온보딩을 어렵게...")
- 수치·규모로 시작 ("3시간 32분", "CPU 100%")
- 문제 넘버링 + 한 줄 부제목

### ④ 해결 여정(Solution Journey)

**"이렇게 해결했습니다"가 아니라 "이 여정을 걸었습니다"**

필수 요소:
- **실패 과정을 숨기지 않는다** — 첫 시도가 바로 성공한 것처럼 쓰지 않음
- **버린 선택지 명시** — 왜 이것을 골랐는지는, 뭘 안 골랐는지를 적어야 완성됨
- **Before/After 대비** — 코드, 아키텍처, 지표를 나란히
- **그래프·메트릭** — "로그가 깔끔해진 그래프" 같은 시각적 증거

### ⑤ 마무리(Reflection)

기술 자랑이 아니라 여정에 대한 의미 부여. "최선의 해법"이 아닌 **"당시 맥락에서의 판단"**으로 표현.
- 시리즈·채용 브랜딩: 오픈엔딩 ("아직 끝나지 않았습니다")
- 단발 글: 구체적 다음 과제 한 줄

## Step 3. 문장 다듬기

### 필요한 정보만 남기기

- 한 문장에 하나의 생각만
- 메타 담화 최소화: "앞서 설명했지만", "다음으로 알아볼 것은" 같은 표현 제거
- 길면 두 문장으로 나눈다

### 구체적으로 쓰기

- **동사 > 명사**: "설정 수행이 필요합니다" X → "설정하세요" O
- **모호한 표현 금지**: "영향을 받을 수도 있습니다" X → "기존 설정이 삭제됩니다" O
- 누가(who), 무엇을(what), 어디에(where), 어떻게(how) 생략하지 않기

### 일관되게 쓰기

- 공식 기술 용어 사용 (대소문자 포함): "k8" X → "Kubernetes" O
- 같은 개념에 여러 단어 혼용 금지: "추가", "첨부", "넣다"를 같은 개념에 섞지 않음
- 약어는 처음 등장 시 풀어쓰기: "SSR(Server-Side Rendering)"
- 외래어는 업계에서 가장 많이 쓰는 표기 따름

### 자연스러운 한국어

- 불필요한 한자어 제거: "수행", "진행", "실시" → 동사로 직접 표현
- 번역체 제거: "API 키를 이용한 사용자 인증 처리가 완료된 후" X → "API 키로 사용자를 인증한 후" O

### 문장의 주체를 분명하게

- 도구·기술을 행동 주체로 쓰지 않기: "이 라이브러리는 DB를 초기화합니다" X → "이 명령어를 실행하면 DB를 초기화할 수 있습니다" O
- 수동형 → 능동형: "설정이 완료되어야 합니다" X → "설정을 완료하세요" O

## 5가지 핵심 신뢰 원칙

| 원칙 | 나쁜 예 | 좋은 예 |
|------|--------|--------|
| **수치로 말한다** | "성능이 크게 개선됐다" | "3시간 32분 → 1분 13초. CPU 100% → 45%" |
| **실패를 숨기지 않는다** | "여러 방법 검토 끝에 선택했다" | "첫 시도는 prefetch 쏠림이 생겨 실패했고..." |
| **버린 선택지 명시** | "최적의 방법으로 해결했다" | "아웃박스 패턴, 별도 API 분리를 검토했으나 기각한 이유는..." |
| **완벽하지 않은 결론** | "최선의 방법으로 구현했다" | "현재 환경에서 가장 단순하고 통제 가능한 선택이라 판단했다" |
| **비즈니스 연결** | "chunk size 100으로 분할 처리" | "chunk 100이 최적임을 테스트로 찾았다. 한 서버에서 돌리면 CPU 100%가 되어 다른 요청이 불가능해진다" |

## 문체와 톤

- **존댓말 대화체**: "~합니다", "~했는데요", "~인데요" (격식체 "~하였다" 금지)
- **1인칭 복수**: "저희 팀이", "우리가" (개인 무용담 금지)
- **독자에게 직접 말 걸기**: "이 숫자가 어떤 의미인지 혹시 짐작이 가실까요?" (글 전체에 2회 이상)
- **비유 적극 활용**: "마치 고고학 발굴처럼...", "블로그 테마를 바꾸듯이..."

핵심 연결 어미: **"~했는데요"** — 단계와 단계 사이를 자연스럽게 연결

## 기술 깊이 기준

대상: "시니어 엔지니어가 다른 팀 시니어에게 설명하는 정도"

- 도메인 용어: 첫 등장 시 한 줄 풀이 (이야기 흐름에 녹임)
- 코드: 핵심 로직만 (전체 코드 X, 의사코드 수준 OK)
- 기술 선택: 반드시 이유와 함께 ("Kubernetes를 도입했습니다" X → "표준화된 방식으로 운영하기 위해 Kubernetes를 도입했습니다" O)
- 수치: 정확도보다 규모감 ("50억 건", "초당 수십만 건")

## 제목 공식

**패턴 A (행위 중심)**: "가맹점은 변함없이, 결제창 시스템 전면 재작성하기"  
**패턴 B (문제 중심)**: "서버 증설 없이 처리하는 대규모 트래픽"

공통: 기술 키워드 + 임팩트 표현. 클릭베이트 금지, 내용을 정직하게 함축.

## 발행 전 체크리스트

**구조 (Step 1)**
- [ ] 문서 유형을 정했는가? (깊은 이해 / 문제 해결 / 학습 / 참조)
- [ ] 개요(Overview)가 제목 바로 아래에 있는가?
- [ ] 가치를 먼저 제공하고 있는가? (배경보다 얻을 것을 먼저)
- [ ] 한 글에서 하나의 핵심 주제만 다루고 있는가?
- [ ] 섹션 이름·순서가 일관적인가?
- [ ] 새 개념 등장 시 즉시 정의했는가?

**5단 구조 (Step 2)**
- [ ] 첫 3문장 안에 훅이 있는가?
- [ ] 배경에 구체적 숫자 1개 이상?
- [ ] 문제 묘사에 실제 코드/구조/데이터 1개 이상?
- [ ] Before/After 대비 최소 1회?
- [ ] 시각 자료(그래프, 다이어그램, 코드 스니펫) 3개 이상?
- [ ] 마지막 단락이 기술 나열이 아닌 회고/의미 부여?

**신뢰도**
- [ ] 첫 시도가 성공한 것처럼 쓰지 않았나?
- [ ] 수치 없이 "개선됐다"만 쓴 곳은 없나?
- [ ] 버린 대안을 이름+이유와 함께 적었나?
- [ ] "당시 맥락에서의 판단"으로 표현했나?
- [ ] 기술명 등장 시 "왜 이것을 썼는지"가 함께 있나?

**문장 (Step 3)**
- [ ] 한 문장에 하나의 생각만 담겼는가?
- [ ] 메타 담화("앞서 설명했듯이", "다음으로")를 제거했는가?
- [ ] 모호한 표현("~할 수도 있다", "일부 경우")을 구체화했는가?
- [ ] 수동형을 능동형으로 바꿨는가?
- [ ] 같은 개념에 일관된 용어를 사용했는가?
- [ ] 약어를 처음 등장 시 풀어썼는가?
- [ ] 불필요한 한자어·번역체를 제거했는가?

**톤**
- [ ] 문장 끝이 격식체("~했다")로 되어 있지 않나?
- [ ] 독자에게 직접 말 거는 문장 2회 이상?
- [ ] 배경 없이 등장하는 기술 용어는 없나?

## 절대 쓰지 않는 표현

- ~를 통해 해결하였다 / 최적의 방법으로 / 성능이 크게 향상됐다
- 다양한 방법을 검토한 결과 / 효율적으로 개선하였다 / 더 나은 방향으로
- 앞서 설명했지만 / 이제 알아보겠습니다 / 결론적으로

## 안티패턴

| 패턴 | 설명 |
|------|------|
| **강의투** | "본 글에서는 ~에 대해 알아보겠습니다" → 바로 이야기에 진입 |
| **기술 자랑** | 왜 도입했는지 없이 "최신 기술 스택을 도입했습니다" |
| **결과만 나열** | 과정 없이 "N% 개선" → 독자가 배울 것이 없음 |
| **추상적 문제** | "코드가 복잡했습니다" → "REFUND 테이블에 계좌이체 취소만 들어가 있었습니다" |
| **감정 과잉** | 고생담 과장 → 담담하되 구체적으로 |
| **가치 뒤로** | 배경·기능 설명 먼저 → 독자가 얻을 것을 먼저 |
| **메타 담화** | "앞서 설명했지만", "다음으로 살펴볼 것은" → 핵심 정보만 |
| **모호한 표현** | "영향을 받을 수도 있습니다" → 정확한 결과를 명시 |
