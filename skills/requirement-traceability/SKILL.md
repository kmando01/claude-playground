# Requirement Traceability

## Overview

과제·명세는 "내가 잘 짠 것"이 아니라 "출제자가 요구한 것"으로 평가된다. 화려한 추가 기능보다 **원본 요구사항 100% 충족**이 합격선에 가깝다. 이 스킬은 원본을 분해해 추적 가능하게 만들고, 산출 직전 빠뜨림을 잡는다.

**핵심 원칙:** 요구사항은 해석하기 전에 먼저 *수집*하라. 빠뜨린 한 줄이 탈락 사유다.

## The Iron Law

```
원본 요구사항에 ID가 없으면, 충족 여부를 증명할 수 없다.
증명할 수 없으면, 충족하지 않은 것으로 간주한다.
```

## When to Use

- 과제 전형(take-home)·RFP·기획서를 받은 직후 (코드 작성 전)
- 요구사항 추적표/매트릭스 작성 시
- 제출·릴리스 직전 최종 검토 시
- "이 기능이 원본에 있던 거였나?" 의심이 들 때

## Process

### Phase 1: 원문 수집 (해석 금지)

원본 문서를 **있는 그대로** 분해한다. 이 단계에서 절대 해석·요약·각색하지 않는다.

```
1. 원본 문서를 모두 나열한다 (기능 / 기술조건 / 제출물 등)
2. 한 문장이 둘 이상의 요구를 담으면 분리한다
   예: "성공 여부와 관계없이 한 번만 참여" → ① 1회 제한 ② 성공/실패 무관
3. 각 요구에 원문을 직접 인용해 붙인다 (근거 추적)
```

CRITICAL: "대충 이런 뜻이겠지"로 요약하면 빠뜨린다. 원문 인용을 반드시 보존하라.

### Phase 2: ID 부여 및 분류

| 분류 | 의미 | 예 |
|------|------|-----|
| **FR** | 명시적 기능 요구 | "선착순 1,000명" |
| **TC** | 기술 제약 (어기면 탈락) | "Spring Boot 3.x", "멀티 인스턴스" |
| **DR** | 제출물 요구 | "README에 API 명세" |
| **IR** | 행간·암묵 (해석으로 도출) | "쿠폰의 실체(유효기간)가 필요" |

CRITICAL: **IR(암묵)을 FR(명시)과 섞지 마라.** 평가자에게 "이건 원본 요구"와 "이건 충족을 위한 내 파생 결정"을 구분해 설명할 수 있어야 한다.

### Phase 3: 키워드 추출 → 개발 함의

원문의 핵심 키워드를 뽑아 개발 의사결정으로 환원한다. 요구를 키워드로 쪼개면 빠뜨림이 드러난다.

```
키워드 | 원문 위치 | 개발 함의 | 연결 요구사항 | 다루는 곳(ADR/정책)
```

자주 놓치는 키워드: "멀티 인스턴스"(→락 전략), "확장성"(→하드코딩 금지), "~와 관계없이"(→경계 조건), "이후에도"(→영속성), "즉시"(→동기 처리).

### Phase 4: 추적표 작성

각 요구사항에 **검증 방법**과 **상태**를 붙인다.

```
| ID | 원문 근거 | 요구사항 | 검증 방법 | 상태 |
```

- 검증 방법은 "테스트 이름 / 코드 위치 / README 섹션"처럼 *실행 가능한 증거*여야 한다
- 상태: 미착수 → 설계됨 → 구현됨 → 테스트됨 → 충족

### Phase 5: 역방향 추적 (YAGNI 점검)

각 *산출물*이 어떤 요구사항을 충족하는지 역으로 매핑한다.

```
| 산출물 | 충족 요구사항 |
```

CRITICAL: 어느 요구사항과도 연결되지 않는 산출물 = **과한 구현(YAGNI 위반)**. 제거하거나 정당화하라.

## Validation Gate (산출 직전 필수)

아래 3단계를 통과하지 못하면 제출/릴리스 보류.

### 단계 1 — 명시 요구 100% 충족
모든 FR/TC/DR이 상태 `충족`이고, 각 행의 검증 방법을 실제 실행해 증거를 기록했는가?

### 단계 2 — 위반 안티패턴 역점검
원본 요구를 깨는 대표 실수를 역으로 확인한다. (도메인별로 작성)
예: "멀티 인스턴스인데 단일 JVM 락 썼나", "즉시 응답인데 비동기로만 처리했나", "한도 초과 가능한 경로가 있나".

### 단계 3 — 자유 결정의 설명 가능성
명세가 "자유 설계하되 설명 가능"을 요구하면, 모든 임의 결정이 근거 문서(ADR)를 갖는가?

| 항목 | 질문 | Yes/No |
|------|------|--------|
| 수집 완전성 | 원본의 모든 문장을 ID로 분해했는가? | |
| 명시/암묵 분리 | FR/TC/DR과 IR을 구분했는가? | |
| 키워드 누락 | 멀티인스턴스·확장성·경계조건 키워드를 잡았는가? | |
| 검증 가능성 | 각 요구에 실행 가능한 검증 방법이 있는가? | |
| 역추적 | 모든 산출물이 요구사항에 연결되는가? (YAGNI) | |
| 최종 게이트 | 제출 전 3단계 검증을 실제 실행했는가? | |

하나라도 No이면 보완 후 마무리한다.

## Red Flags

| 생각 | 현실 |
|------|------|
| "요구사항은 대충 알겠으니 바로 코드" | 빠뜨린 한 줄이 탈락 사유다 |
| "이 기능 넣으면 멋있겠다" | 원본에 없으면 가산점 아니라 노이즈 |
| "재조회는 당연히 되겠지" | 영속성을 설계 안 하면 TTL 만료 후 깨진다 |
| "테스트는 해피패스만" | 명세가 "예외/경계 조건 검증"을 요구했다면 미충족 |
| "제출 전에 한 번 보면 되지" | 게이트 없이 보면 항상 빠뜨린다. 체크리스트로 강제하라 |

## Common Mistakes

| 실수 | 결과 | 방지 |
|------|------|------|
| 원문을 요약해서 정리 | 미묘한 요구 누락 | Phase 1 원문 인용 보존 |
| 암묵 요구를 명시처럼 제시 | 평가자에게 "출제 안 한 걸 했다"는 인상 | Phase 2 IR 분리 |
| 검증 방법이 추상적 | 충족 증명 불가 | Phase 4 실행 가능한 증거 |
| 추가 기능 자랑 | 원본 충족 부족을 가림 | Phase 5 역추적 |
| 제출 직전 눈으로만 검토 | 빠뜨림 반복 | Validation Gate 강제 실행 |
