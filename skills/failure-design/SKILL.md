# Failure Design

## Overview

Happy path만 설계하면 운영에서 터진다. 성공 흐름 옆에 실패 흐름을 반드시 같이 설계한다.

**핵심 원칙:** 성공 규모를 추정했으면, 실패 규모도 추정하라. 데이터 흐름을 그렸으면, 에러 흐름도 그려라.

## The Iron Law

```
성공 흐름만 있는 설계는 미완성이다.
```

아래 3가지가 모두 포함되지 않으면 설계를 완료로 간주하지 않는다:
1. 실패 규모 추정
2. 에러 흐름 다이어그램
3. 인터페이스 실패 모드 분석

## When to Use

- 기능 설계 / ADR 작성 시
- 규모 추정(TPS, 일 처리량) 시
- 외부 시스템 연동 설계 시
- Port/Adapter 인터페이스 정의 시
- 아키텍처 문서 작성 시

## Process

기존 설계 흐름에 3단계를 병행한다. 독립 프로세스가 아니라, 설계의 각 단계에 끼워넣는 체크포인트다.

### Phase 1: 실패 규모 추정

**WHEN:** 성공 규모 추정을 마친 직후

성공 규모만 추정하면 "일 30M건 처리" 같은 숫자만 남는다. 실패 규모를 같이 추정하면 에러 분류, 사전 검증, resilience 전략이 자연스럽게 도출된다.

**실패 추정 템플릿:**

```
총 처리량: 일 ___건, 피크 ___TPS

실패율 가정: __% (외부 API 기준 1~5% 일반적)
  → 일 실패 건수: ___건

실패 유형 분류:
  재시도 가능 (일시적 장애): __% → 일 ___건
    예: 503, 429, timeout, connection reset
  재시도 불가능 (영구 실패): __% → 일 ___건
    예: 400, 인증 실패, 데이터 오류

재시도 불가능 건 중 사전 차단 가능: __%
  → 입력 검증으로 막을 수 있는 것은?
  → 남은 건 (코드 버그/스펙 변경): 일 ___건 → DLQ + 모니터링
```

CRITICAL: 숫자가 채워지지 않으면 다음 단계로 진행하지 않는다.

### Phase 2: 에러 흐름 다이어그램

**WHEN:** 데이터 흐름(Happy Path)을 그린 직후

데이터 흐름의 각 boundary(경계)에서 "여기서 실패하면?" 질문을 던진다.

**작성 방법:**

Happy Path의 각 단계마다:
1. 이 단계에서 발생 가능한 실패는?
2. 그 실패는 어디로 가는가? (응답 에러? 재시도? DLQ?)
3. 사전에 막을 수 있는가?

**에러 흐름 템플릿:**

```
[입력 단계]
  → 검증 실패 → 즉시 에러 응답 (여기서 끝)
  → 검증 통과 → [처리 단계]
    → 내부 장애 (DB 등) → 에러 응답 / 재시도
    → 성공 → [외부 호출 단계]
      → 일시적 실패 (503/429/timeout) → App Retry → Infra Retry → DLQ
      → 영구 실패 (400/파싱 에러) → 즉시 DLQ (retry 건너뜀)
      → 대량 장애 (CB OPEN) → 소비 중단 → 자동 복구 대기
      → 성공 → 완료
```

CRITICAL: Happy Path 다이어그램만 있고 에러 흐름이 없으면 설계 미완성이다.

### Phase 3: 인터페이스 실패 모드 분석

**WHEN:** Port 인터페이스나 외부 연동 인터페이스를 정의할 때

인터페이스를 정의하면 실패가 추상화 뒤에 숨는다. `send() → SendResult(success/fail)`처럼 모든 실패를 하나로 뭉치면, 호출자가 적절히 대응할 수 없다.

**인터페이스마다 이 질문을 반드시 수행:**

```
1. 이 인터페이스의 구현체가 실패할 수 있는 방법은 몇 가지인가?
   → 나열하라

2. 그 실패들은 성격이 같은가, 다른가?
   → 재시도 가능 vs 불가능 구분

3. 호출자가 실패 유형을 구분할 수 있는가?
   → 반환 타입이 유형 구분을 지원하는가?
   → 지원하지 않으면 인터페이스 수정 필요

4. 실패를 줄이기 위해 입력 단계에서 검증할 수 있는 것은?
   → "Consumer에서 발견할 에러를 API에서 미리 잡을 수 있는가?"

5. 대량 실패 시 시스템 전체에 미치는 영향은?
   → CB OPEN 시 메시지 폭주? 연쇄 장애? 리소스 고갈?
```

CRITICAL: 질문 4를 건너뛰면 "API에서 잡을 수 있는 에러가 Consumer까지 흘러가는" 문제가 반복된다.

## Validation Gate

설계 완료 전 최종 점검. 아래 항목이 모두 Yes여야 설계 완료:

| 항목 | 질문 | Yes/No |
|------|------|--------|
| 실패 규모 | 실패율과 유형별 건수를 추정했는가? | |
| 에러 흐름 | 각 boundary에서 실패 시 경로를 그렸는가? | |
| 사전 검증 | 입력 단계에서 막을 수 있는 에러를 식별했는가? | |
| 실패 분류 | 재시도 가능/불가능 에러를 구분했는가? | |
| 인터페이스 | 반환 타입이 실패 유형 구분을 지원하는가? | |
| 대량 장애 | CB OPEN 등 대량 실패 시 시스템 동작을 정의했는가? | |
| 예외 소유권 | 각 예외의 catch 위치와 전파 경로를 명시했는가? | |
| 에러 매트릭스 | 예외 타입별 Retry/CB/Kafka 동작을 표로 정리했는가? | |
| DLQ 후처리 | 자동 복구 불가능한 건의 처리 방안이 있는가? | |

하나라도 No이면 해당 항목을 보완한 후 설계를 마무리한다.

## Phase 4: 예외 소유권 분석

**WHEN:** 에러 흐름 설계 완료 후, 구현 계획 작성 전

**핵심 원칙:** 예외를 어디서 잡느냐가 곧 아키텍처 결정이다.

내 코드는 혼자 실행되는 게 아니라, 여러 프레임워크 레이어를 관통하는 파이프라인의 한 지점이다. 예외를 catch하는 순간, 그 위에 있는 레이어(Retry, CircuitBreaker, Kafka ErrorHandler 등)가 작동할 기회를 빼앗는다.

**각 예외에 대해 반드시 이 질문을 수행:**

```
1. 이 예외의 최종 처리 책임자는 누구인가?
   → 내 코드? Retry 레이어? CB 레이어? Kafka ErrorHandler?

2. 내가 여기서 catch하면 상위 레이어가 잃는 것은?
   → 재시도 기회? CB 카운트? DLQ 라우팅?

3. 값(SendResult)으로 변환하면 vs 예외로 전파하면?
   → 값 변환: Kafka는 "성공 소비"로 간주 → retry 불가
   → 예외 전파: Kafka ErrorHandler가 retry/DLQ 판단 가능

4. 예외 타입별 전파 경로를 명시했는가?
   → catch 순서가 전파 경로를 결정한다
   → 서브클래스를 먼저 catch하지 않으면 상위 catch에 삼켜진다
```

**에러 분류 매트릭스 (구현 계획에 반드시 포함):**

```
| 예외 타입          | catch 위치    | App Retry | CB 카운트 | Kafka Retry |
|-------------------|--------------|-----------|----------|-------------|
| 400 Bad Request   | callApi()    | X         | X        | X (DLQ 직행) |
| 429 Too Many Req  | 전파 (catch X)| O         | O        | O           |
| 500 Server Error  | 전파 (catch X)| X         | O        | O           |
| 503 Unavailable   | 전파 (catch X)| O         | O        | O           |
| Timeout/ConnReset | 전파 (catch X)| X         | O        | O           |
| CB OPEN           | ???          | X         | -        | ???         |
```

CRITICAL: 매트릭스의 "catch 위치" 열이 비어있으면 구현 시 "편의상 전부 catch" 패턴이 반복된다. **catch하는 것과 전파하는 것을 설계 시점에 결정하라.**

CRITICAL: CB OPEN 시 `CallNotPermittedException`의 처리 방식도 명시하라. 값으로 변환하면 해당 메시지는 영구 실패, 예외로 전파하면 Kafka retry 가능.

## Red Flags

이런 생각이 들면 STOP:

| 생각 | 현실 |
|------|------|
| "실패 케이스는 나중에 생각하자" | 나중은 운영 장애 후다 |
| "Port 인터페이스가 알아서 처리할 거야" | 인터페이스는 실패를 숨길 뿐이다 |
| "외부 API는 거의 안 실패한다" | 1%도 일 30만 건이다 |
| "재시도하면 되니까 괜찮다" | 400은 100번 재시도해도 400이다 |
| "Happy Path 먼저 구현하고 에러 처리 추가하자" | 에러 처리가 아키텍처를 바꿀 수 있다 |
| "Mock이라 실패 안 한다" | Mock은 임시. 실제 연동 시 설계가 없으면 처음부터 다시 한다 |
| "예외가 위험하니 catch로 다 잡아서 안전하게 값으로 감싸자" | 예외를 잡는 순간 상위 레이어(Retry, CB, ErrorHandler)의 판단 기회를 빼앗는다 |
| "SendResult로 통일하면 깔끔하다" | 프레임워크 레이어는 예외 타입을 보고 판단한다. 값으로 바꾸면 신호가 사라진다 |

## Common Mistakes

| 실수 | 결과 | 방지 |
|------|------|------|
| 성공 규모만 추정 | 에러 분류/사전 검증 누락 | Phase 1 실패 추정 병행 |
| 데이터 흐름만 설계 | 장애 시 메시지 유실 | Phase 2 에러 흐름 병행 |
| 인터페이스 실패를 하나로 뭉침 | 호출자가 대응 불가 | Phase 3 실패 모드 분석 |
| 사전 검증 가능한 에러를 후단에서 발견 | DLQ 적체, 수동 복구 | "입력에서 막을 수 있는가?" 질문 |
| 대량 장애 시나리오 미고려 | CB OPEN → 메시지 폭주 → 전체 장애 | Phase 3 질문 5 수행 |
| 예외를 전부 catch하여 값으로 변환 | 상위 레이어(Retry, CB, ErrorHandler)가 작동 불가 | Phase 4 예외 소유권 분석 |
| catch 순서 미고려 | 서브클래스(429)가 상위 catch(HttpClientErrorException)에 삼켜짐 | Phase 4 catch 순서 = 전파 경로 |
| CB OPEN 시 예외 처리 미정의 | 이미 폴링된 메시지가 영구 FAILED | Phase 4 매트릭스에 CB OPEN 행 필수 |
